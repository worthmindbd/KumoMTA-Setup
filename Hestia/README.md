# Hestia -> KumoMTA: enable Exim AUTH PLAIN for authenticated relay

These helper files live with the KumoMTA installer for safekeeping, but they
are applied on the **HestiaCP server**, not the KumoMTA server.

## Why this is needed

KumoMTA supports SMTP **AUTH PLAIN only**. Attempting any other mechanism
returns:

```
504 5.5.4 AUTH LOGIN not supported
```

HestiaCP's stock Exim relay authenticator (`smtp_relay_login`) uses
**AUTH LOGIN**. So when Hestia relays outbound mail to a KumoMTA smart host
with a username/password, authentication fails (`535 AUTH invalid` /
`504 ... AUTH LOGIN not supported`) and mail never relays.

The fix adds a second authenticator, `smtp_relay_plain` (PLAIN), to Hestia's
Exim template **without removing** the LOGIN block. Exim, acting as a client,
uses whichever mechanism the smart host advertises -- KumoMTA advertises only
PLAIN, so Hestia uses PLAIN; a LOGIN-only smart host still works via the
original block.

## Files

| File                          | Install location (HestiaCP server)            | Purpose |
|-------------------------------|-----------------------------------------------|---------|
| `hestia-exim-plain-fix.sh`    | `/usr/local/sbin/hestia-exim-plain-fix.sh`    | Idempotently adds the PLAIN authenticator, rebuilds + reloads Exim. |
| `99-hestia-exim-plain-fix`    | `/etc/apt/apt.conf.d/99-hestia-exim-plain-fix`| APT hook: re-applies the fix after package updates (self-healing). |

## Install (on the HestiaCP server)

```bash
# 1. Back up the current Exim template
cp /etc/exim4/exim4.conf.template /etc/exim4/exim4.conf.template.bak-$(date +%F)

# 2. Copy the script into place and make it executable
#    (paste/scp hestia-exim-plain-fix.sh to this path first)
chmod +x /usr/local/sbin/hestia-exim-plain-fix.sh

# 3. Install the APT hook (paste/scp 99-hestia-exim-plain-fix to this path)
#    /etc/apt/apt.conf.d/99-hestia-exim-plain-fix

# 4. Apply now
/usr/local/sbin/hestia-exim-plain-fix.sh
```

## Verify

```bash
# Block present in the LIVE (compiled) config -- must print 1
grep -c smtp_relay_plain /var/lib/exim4/config.autogenerated

# Exim config still parses cleanly
exim4 -bV

# Service healthy
systemctl status exim4 --no-pager
```

## Then, in the HestiaCP panel

Set the SMTP Relay per mail domain (Mail -> domain -> Edit -> SMTP Relay):

- Host: `smtp.<your-kumomta-domain>`  (e.g. `smtp.neumannassociatesnews.io`)
- Port: `587`
- Username: the KumoMTA SMTP user (e.g. `news@<domain>`)
- Password: the value stored in KumoMTA's `/opt/kumomta/etc/auth.db`
  (printed in `/root/kumomta-install-summary.txt` on the KumoMTA server)

The password in the panel MUST match KumoMTA's `auth.db` exactly, or you get
`535 AUTH invalid`.

## Notes / gotchas

- **Per-domain relay settings** are stored by Hestia in
  `/etc/exim4/domains/<domain>/smtp_relay.conf` (user data) and are NOT
  overwritten by updates. Only the authenticator (template) needs the
  self-healing hook.
- **Retry timer:** after a failed attempt, Exim defers with
  "retry time not reached". Force a retry with `exim -qff` (whole queue) or
  `exim -M <message-id>` (one message).
- **DKIM on this path:** Hestia's authenticated relay transport
  (`smtp_relay_smtp`) does NOT add a DKIM signature -- KumoMTA signs instead.
  For each ADDITIONAL sending domain, add its key to KumoMTA's
  `dkim_data.toml`, or that domain's mail goes unsigned.
- **SPF for new domains:** add `include:smtp.<your-kumomta-domain>` (or the
  KumoMTA sending-IP range) to each new sending domain's SPF record.
